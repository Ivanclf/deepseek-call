# Deepseek 调用页面需求文档

## 目标
设计并实现一个页面，用于调用 Deepseek 接口，实现对话式问答功能。页面应具备历史对话管理、流式返回展示、以及默认开启深度思考功能。

## 功能需求

### 1. 对话历史管理
- 对话历史以 JSON 格式存储在 `history` 文件夹中。
- 每个独立对话对应一个 JSON 文件，文件名可使用对话 ID、时间戳或用户指定的标识。
- JSON 结构示例：
  - `conversationId`
  - `createdAt`
  - `messages`: 对话消息数组，每条消息包含 `role`、`content`、`timestamp` 等字段。
- 每次发送新请求时，需要将当前对话对应的全部历史消息读取并包装进 Deepseek 请求内容中。
- 发送后的消息也应追加到对应对话 JSON 文件中，保持历史一致性。

### 2. 接口调用逻辑
- 页面负责构造 Deepseek 请求体，包含：
  - 当前用户输入问题
  - 对话历史消息列表
  - 深度思考功能默认开启的标志位（例如 `enableDeepThinking: true`）
- 不需要实现联网功能。请求逻辑可以模拟 Deepseek API 调用，或在本地封装出接口请求结构与返回流程。
- 即便不联网，也应保留接口请求与返回的抽象层，便于未来接入真实 Deepseek 服务。

### 3. 流式返回展示
- 对 Deepseek 响应进行流式渲染，页面应能逐步展示返回内容。
- 页面应同时展示 Deepseek 的思考过程文本，先显示思考过程再显示最终回答。
- 即便是本地模拟返回，也应采用分块或逐步追加方式渲染，体现“流式返回”效果。
- 页面应显示当前请求状态，例如 `正在请求`、`接收中`、`完成` 等。

## 页面交互需求
- 输入框：用于用户输入问题。
- 发送按钮：触发请求，并将当前对话历史与新输入一起发送。
- 对话区：展示历史消息与本次会话内容，区分用户与 Deepseek 回复。
- 流式输出区：用于展示 Deepseek 的逐步返回结果。
- 对话切换/新建功能：支持选择或创建新的对话，并在 `history` 文件夹中生成对应 JSON 文件。

## 数据组织
- `history/` 文件夹结构：
  - `history/{conversationId}.json`
- 每个对话 JSON 包含示例结构：
```json
{
  "conversationId": "conversation-20260531-001",
  "createdAt": "2026-05-31T12:00:00Z",
  "messages": [
    {
      "role": "user",
      "content": "你好，Deepseek！",
      "timestamp": "2026-05-31T12:01:00Z"
    },
    {
      "role": "assistant",
      "content": "你好！我能帮你什么？",
      "timestamp": "2026-05-31T12:01:20Z"
    }
  ]
}
```

## 非功能需求
- 系统不依赖外网，不需要真实联网能力。
- 深度思考功能默认开启，页面内部请求结构或模拟实现应体现该配置。
- 对话历史持久化存储，刷新页面后仍可继续当前对话。
- 设计应便于后续扩展为真实 Deepseek 接口调用。

## 交付内容
- 一个实现 Deepseek 调用页面的需求说明文档（`requirement.md`）。
- 说明文档应明确历史对话管理、默认深度思考、流式返回三个核心需求。
